# 从代码到知识：Andrej Karpathy 的 LLM 知识库方法论

**作者**：时光的沙盒  
**来源**：[知乎专栏](https://zhuanlan.zhihu.com/p/2023803203725135995)

---

> 当 AI 大牛开始用 LLM 管理知识而非编写代码，这标志着个人知识管理范式的根本性转变。

2026年4月3日，AI 领域最具影响力的教育家之一 Andrej Karpathy 在 X 平台分享了他最新的工作流：使用 LLM 构建个人知识库（Personal Knowledge Base）。这篇看似简短的推文，实则揭示了一个正在发生的深刻转变——LLM 正在从"代码生成工具"进化为"知识管理伙伴"。

Karpathy 坦言，他近期的大部分 token 吞吐量已不再用于代码操作，而是用于知识操作（ manipulating knowledge）——以 Markdown 和图片的形式存储、整理、关联和提炼信息。这一转变不仅关乎工具的使用，更代表着一种全新的知识工作哲学。

## 一、核心思想：从"写代码"到"管理知识"的范式转移

### 1.1 什么是 LLM 知识库？

Karpathy 描述的 LLM 知识库是一个由 AI 自主构建和维护的维基系统。其核心架构可以概括为：

```text
原始数据（raw/）→ AI 编译（compilation）→ 维基（wiki/）→ 多模态输出 → 持续增强
```

**关键特征：**

- **AI 主导**：LLM 负责编写和维护所有 wiki 内容，人类极少直接编辑
- **增量编译**：源文档被逐步"编译"成结构化的知识网络
- **双向链接**：自动生成反向链接（backlinks），构建概念关联图谱
- **多模态融合**：支持文本、图片、代码库、数据集等多种数据类型
- **自我增强**：查询结果会回流到知识库，形成正反馈循环

### 1.2 为什么是"知识操作"而非"代码生成"？

Karpathy 的这一转变具有标志性意义。作为前 Tesla AI 总监、OpenAI 创始成员，他一直是 AI 辅助编程的倡导者。但现在，他将 LLM 的注意力转向了知识管理：

> "最新一代的 LLM 在这方面表现得相当出色。" —— Andrej Karpathy

这种转变反映了三个深层趋势：

1. **代码生成已趋于成熟**：GitHub Copilot、Cursor、Claude Code 等工具已经很好地解决了编程辅助问题
2. **知识爆炸需要新范式**：信息过载时代，人类需要 AI 帮助筛选、整理和关联海量信息
3. **长上下文窗口的突破**：Claude 4.6 Opus（100万 token）、Gemini 3.0 Pro（100万 token）等模型让"一次性理解整个知识库"成为可能

## 二、Karpathy 工作流深度解析：五个层次的方法论

Karpathy 的推文虽然简短，但蕴含了一个完整的工作流体系。我们可以将其拆解为五个层次：

### 2.1 数据摄取层（Data Ingest）

**核心任务**：将分散的源文档统一索引到 raw/ 目录

**支持的源类型：**

- 文章（网页、博客、新闻）
- 学术论文
- 代码仓库
- 数据集
- 图片
- 任何有价值的信息源

**工具链：**

- Obsidian Web Clipper：浏览器扩展，一键将网页转为 Markdown
- 热键脚本：自动下载相关图片到本地，确保 LLM 可以引用
- PDF 转换器：将学术论文转为 Markdown（可使用 Claude Code 的 PDF-to-Markdown skill）

**关键洞察**：Karpathy 强调将图片也纳入知识库，因为视觉信息往往包含文本无法表达的关系和模式。这与多模态 LLM 的发展趋势高度一致。

### 2.2 编译层（Compilation）：AI 构建维基

这是整个工作流的核心创新点。Karpathy 使用 LLM 将原始数据"编译"成结构化的 wiki：

**编译过程包括：**

1. 摘要生成：为每份 raw 文档创建摘要
2. 反向链接：自动识别文档间的引用关系
3. 概念分类：将数据归类到概念体系
4. 文章撰写：为每个概念撰写专题文章
5. 链接网络：构建概念间的超链接网络

**输出格式：**

- 纯 Markdown 文件（.md）
- 目录结构化的文件夹体系
- 支持 YAML frontmatter 的元数据

**技术实现**： Karpathy 提到这是通过"一系列脚本"（a hacky collection of scripts）实现的。社区已经出现了更成熟的解决方案：

- obsidian-claude（GitHub）：31 个命令、27 个代理、19 个 AI 技能的完整研究流程
- Claude Code Skills：可复用的工作流指令，支持 Markdown 优化、文档转换等
- PARA 方法：Projects, Areas, Resources, Archives 的组织框架

### 2.3 IDE 层：Obsidian 作为前端

Karpathy 选择 Obsidian 作为知识库的"IDE 前端"，这是一个深思熟虑的选择：

**Obsidian 的核心优势：**

1. **纯文本存储**：所有数据都是本地 Markdown 文件，无供应商锁定
2. **双向链接**：`[[wiki link]]` 语法天然支持知识网络构建
3. **图谱视图**：Graph view 可视化知识关联
4. **插件生态**：丰富的社区插件扩展功能
5. **本地优先**：数据完全由用户掌控，隐私安全

**关键插件：**

- Marp：将 Markdown 转为幻灯片
- Dataview：基于元数据的动态查询
- Copilot：AI 辅助写作和问答
- Auto Note Mover：基于标签的自动归档

**重要原则**： Karpathy 特别强调——**你几乎从不手动编写或编辑 wiki，这是 LLM 的领域**。Obsidian 在这里主要作为**查看器**（viewer）和**导航器**（navigator），而非编辑器。

### 2.4 问答层（Q&A）：与知识库对话

当 wiki 达到一定规模（Karpathy 的例子：~100 篇文章，~40 万字），真正的魔法开始显现：

**能力跃迁：**

- 可以向 LLM 代理提出复杂问题
- LLM 会自主"研究"答案，跨文档检索信息  
- 无需复杂的 RAG 管道，依靠 LLM 的自动索引维护

**Karpathy 的发现：**

> "我曾以为必须用上花哨的 RAG，但 LLM 在自动维护索引文件和文档摘要方面表现得相当好，在这个'小规模'下能轻松读取所有重要的相关数据。"

这揭示了一个重要趋势：随着上下文窗口的扩大，简单的"全上下文加载"可能比复杂的 RAG 更有效——至少在个人知识库的尺度上（<100万 token）。

### 2.5 输出层：多模态知识呈现

Karpathy 不喜欢在终端获得纯文本答案，而是让 LLM 生成：

- **Markdown 报告**：结构化文档
- **幻灯片（Marp 格式）**：演示文稿
- **Matplotlib 图像**：数据可视化
- **其他格式**：取决于查询需求

**关键实践：**

> "我经常将这些输出'归档'回 wiki，以增强进一步查询的能力。因此，我自己的探索和查询总是能在知识库中'累加'。"

这形成了一个**自我增强的飞轮**：查询 → 生成洞察 → 归档 → 丰富知识库 → 支持更深查询。

### 2.6 健康检查层（Linting）：持续优化

Karpathy 还运行 LLM "健康检查"来维护 wiki 质量：

- **不一致检测**：发现矛盾或过时的信息
- **缺失数据填补**：通过网络搜索补充信息
- **新文章候选**：发现有趣的连接，建议新主题
- **数据完整性**：增量清理和增强

LLM 还能建议进一步的问题，引导用户探索未知领域。

## 三、技术选型：RAG vs 长上下文窗口

Karpathy 的方法论引发了一个关键问题：在个人知识库场景下，RAG（检索增强生成）和长上下文窗口，哪个更优？

### 3.1 RAG：经典方案

**工作原理：**

1. 文档切分成 chunks
2. 用 embedding 模型转为向量
3. 存储到向量数据库
4. 查询时检索最相关的 chunks
5. 将 chunks + 问题送入 LLM

**优势：**

- **成本可控**：只处理相关片段，token 消耗低
- **速度快**：检索通常在 50-200ms 完成
- **可扩展**：支持 TB/PB 级数据
- **来源可追溯**：能显示引用的具体文档

**劣势：**

- **架构复杂**：需要向量数据库、embedding 模型、分块策略
- **信息碎片化**：可能丢失跨文档的全局关联
- **调参困难**：chunk 大小、重叠度、检索数量等参数敏感

### 3.2 长上下文窗口：新兴范式

**2025-2026 年的突破性进展：**

| 模型 | 上下文窗口 | 关键特性 |
|------|-----------|----------|
| Claude 4.6 Opus | 100万 token | 复杂推理、编码、agentic 工作流 |
| Gemini 3.0 Pro | 100万 token | 多模态、100% recall（53万 token 内） |
| GPT-5.2 | ~100万 token | 编码和长上下文推理优化 |

**优势：**

- **架构极简**：无需 RAG 管道，直接全量加载
- **全局理解**：能捕捉跨文档的隐含关联
- **实现快速**：几分钟即可搭建原型
- **推理深度**：支持复杂的跨文档分析

**劣势：**

- **成本高昂**：100万 token 输入可能花费 $3-15
- **延迟较高**：处理大上下文需要 30-60 秒
- **注意力稀释**：关键信息可能被"淹没"在中间（lost in the middle 问题）

### 3.3 Karpathy 的选择与社区实践

Karpathy 明确选择了长上下文窗口路线：

> "我曾以为必须用上花哨的 RAG，但 LLM 在自动维护索引文件...能轻松读取所有重要的相关数据。"

**适用场景对比：**

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 个人知识库（<100万 token） | 长上下文 | 简单、全局理解、成本低（数据量小） |
| 企业知识库（>1000万 token） | RAG | 可扩展、成本可控、权限管理 |
| 动态数据（频繁更新） | RAG | 增量更新容易，无需重新加载全量 |
| 静态档案（历史文档） | 长上下文 | 一次性加载，深度分析 |
| 多模态混合 | 混合架构 | RAG 检索 + 长上下文推理 |

### 3.4 混合架构：最佳实践

社区普遍推荐的混合方案：

1. **RAG 做初筛**：从海量文档中检索最相关的 5-10 篇
2. **长上下文做推理**：将这 5-10 篇完整文档一次性送入 LLM
3. **层次化索引**：维护自动生成的摘要和索引文件

这样既保留了 RAG 的可扩展性，又获得了长上下文的全局理解能力。

## 四、国内生态实践：飞书、扣子与本土化工具链

Karpathy 的方案基于 Obsidian + Claude，但在国内生态中，我们有更多本土化选择。

### 4.1 飞书多维表格：零代码 AI 知识库

**核心能力**： 飞书多维表格在 2026 年已经进化为强大的 AI 自动化平台：

- **AI 字段捷径**：内置豆包生图、DeepSeek 文本生成等 AI 能力
- **自动化流程**："当满足条件时 → 执行动作"的零代码工作流
- **多维视图**：表格、看板、日历、甘特图等多种视图
- **权限管理**：精细的分享和权限控制

**知识库构建流程：**

1. **数据录入：**

   - 手动输入或批量导入（Excel/CSV）
   - 飞书剪存：浏览器插件一键保存网页
   - API 接入：从其他系统自动同步

2. **AI 处理：**

   - 使用 AI 字段自动生成摘要、标签、分类
   - 配置自动化规则："当有新记录时 → 调用 DeepSeek API 生成摘要 → 更新字段"

3. **知识组织：**

   - 使用"关联"字段建立文档间连接
   - 通过"分组"和"筛选"构建分类体系
   - 利用"看板视图"进行主题管理

4. **问答与输出：**

   - 飞书智能伙伴（Feishu Copilot）支持基于多维表格的问答
   - 可以生成报告、发送通知、创建任务

**成本优势：**

- 飞书提供免费额度，可零成本启动
- 国内网络环境，无需翻墙
- 与企业现有工作流无缝集成

### 4.2 扣子（Coze）+ Dify：AI 智能体工作流

**平台定位：**

- **扣子（Coze）**：字节跳动推出的 AI 应用开发平台
- **Dify**：开源的 LLM 应用开发框架，支持本地部署

**2026 年的关键更新：**

1. **知识库定制：**
   - 支持结构化数据（FAQ、产品手册、技术文档）
   - 文档预处理：自动分段、向量化、索引
   - 分层知识策略：通用知识 + 业务专属知识

2. **工作流设计：**
   - 可视化拖拽配置
   - 支持触发条件、API 调用、多轮对话
   - 与 ERP、CRM 等系统深度集成

3. **本地部署：**
   - 2026 年新规强调数据安全
   - 支持私有化部署，符合金融、医疗、政务合规要求
   - 兼容国产大模型（通义千问、文心一言）

**搭建步骤：**

```text
1. 知识库建设 → 2. 工作流设计 → 3. 本地部署与训练
```

**典型应用场景：**

- 电商客服智能体
- 企业内部知识问答
- 自媒体内容生成
- 数据分析与报告

### 4.3 Notion AI：国际化团队的替代方案

对于习惯使用 Notion 的用户，Notion AI 提供了类似的能力：

**核心功能：**

- **Q&A 功能**：跨页面、数据库、文档的智能搜索
- **数据库属性自动填充**：AI 自动分类、标签、摘要
- **会议笔记自动化**：生成摘要、提取行动项、创建任务
- **知识库结构生成**：从非结构化信息创建分类体系

**定价：**

- $10/成员/月（附加在 Notion 订阅之上）
- 对大型团队成本较高，但个人使用性价比高

## 五、进阶探索：合成数据生成与模型微调

Karpathy 在推文中提到了一个前沿方向：

> "随着仓库增长，自然的想法是考虑合成数据生成 + 微调，让你的 LLM 将数据'知道'在权重中，而非仅依赖上下文窗口。"

### 5.1 为什么要微调？

**当前方案的限制：**

- **上下文窗口仍有限**：即使 100万 token，也无法容纳终身知识
- **每次查询都需加载**：成本高、延迟大
- **无法离线使用**：依赖 API 调用

**微调的优势：**

- **知识内化**：将知识"压缩"到模型权重中
- **即时响应**：无需加载上下文，推理速度快
- **离线可用**：本地运行，隐私安全
- **成本降低**：推理时 token 消耗大幅减少

### 5.2 合成数据生成策略

**核心挑战**：如何生成高质量的微调数据？

**最新研究进展（2025-2026）：**

1. **主动合成数据生成（Active Synthetic Data Generation）：**
   - 学生模型引导数据生成
   - 优先选择"困难样本"（高不确定性数据）
   - 迭代优化：生成 → 微调 → 评估 → 再生成

2. **教师-学生范式：**
   - 使用大模型（如 Claude 4 Opus）作为教师
   - 生成问答对、推理链、解释文本
   - 小模型（如 Qwen3-32B）作为学生进行微调

3. **数据选择策略：**
   - **高损失样本**：模型预测损失高的数据
   - **低奖励样本**：外部奖励模型评分低的数据
   - **错误答案样本**：模型当前回答错误的问题

### 5.3 微调技术路线

**2025 年的实用方案：**

| 方法 | 适用场景 | 资源需求 |
|------|---------|----------|
| LoRA | 快速实验、消费级 GPU | 1x RTX 4090 |
| QLoRA | 量化微调、更低显存 | 1x RTX 3090 |
| 全参数微调 | 最大效果、企业级 | 8x A100 |
| 蒸馏 | 从大模型迁移知识 | 中等算力 |

**开源工具链：**

- **Unsloth**：2倍速微调，显存减少 70%
- **Axolotl**：YAML 配置，一键微调
- **Llama-Factory**：Web UI，支持 100+ 模型
- **Firefly**：中文场景优化

### 5.4 从知识库到专属模型：完整流程

```text
知识库（Markdown）
    ↓
合成数据生成（LLM 生成问答对）
    ↓
数据清洗与筛选（选择高质量样本）
    ↓
LoRA 微调（本地或云端）
    ↓
评估与迭代（测试 → 改进）
    ↓
部署专属模型（API 或本地运行）
```

**预期效果：**

- 经过微调的 7B 模型，在特定领域可能接近甚至超过未微调的 70B 模型
- 推理成本降低 10-100 倍
- 响应延迟从秒级降至毫秒级

## 六、实践指南：如何开始构建你的 AI 知识库

基于以上分析，这里提供一个分阶段的实施路线图。

### 6.1 第一阶段：基础搭建（1-2 周）

**目标**：建立最小可行系统（MVP）

**步骤 1：选择工具栈**

根据个人/团队情况选择：

**方案 A：Karpathy 原版（技术向）**

- Obsidian（知识库前端）
- Claude Code / Claude Desktop（AI 引擎）
- Obsidian Web Clipper（数据摄取）

**方案 B：国内零代码（业务向）**

- 飞书多维表格（数据 + 自动化）
- DeepSeek / 豆包（AI 能力）
- 飞书剪存（数据摄取）

**步骤 2：建立目录结构**

```
my-knowledge-base/
├── raw/                    # 原始数据
│   ├── articles/          # 文章
│   ├── papers/            # 论文
│   ├── images/            # 图片
│   └── code/              # 代码片段
├── wiki/                  # AI 编译的维基
│   ├── concepts/          # 概念文章
│   ├── summaries/         # 摘要
│   └── index.md           # 索引
├── inbox/                 # 待处理
├── output/                # 生成的输出
│   ├── reports/           # 报告
│   ├── slides/            # 幻灯片
│   └── visuals/           # 可视化
└── scripts/               # 自动化脚本（可选）
```

**步骤 3：配置 AI 工作流**

**Obsidian + Claude 方案：**

- 安装 obsidian-claude 插件
- 配置 Claude Code skill 文件
- 创建模板：摘要模板、概念文章模板、索引模板
- 设置热键：一键摄取网页、一键生成摘要

**飞书方案：**

- 创建多维表格
- 添加字段：标题、内容、AI 摘要、标签、来源、创建时间
- 配置自动化："当有新记录时 → 调用 DeepSeek API → 更新 AI 摘要字段"
- 安装飞书剪存浏览器插件

**步骤 4：开始摄取数据**

- 每天摄取 3-5 篇高质量文章
- 优先关注你的核心领域
- 不要囤积，立即处理（inbox zero 原则）

### 6.2 第二阶段：自动化与优化（2-4 周）

**目标**：建立自动化工作流，减少人工干预

**步骤 1：自动化摄取**

- 配置 RSS 订阅自动导入
- 设置邮件转发到知识库
- 使用 Zapier / Make 连接其他应用

**步骤 2：AI 编译流程**

创建标准化的 AI 提示词模板：

```markdown
# 知识编译提示词

你是一位知识管理专家。请处理以下原始数据：

1. 生成结构化摘要（3-5 个要点）
2. 提取关键概念，链接到已有概念
3. 识别与现有文章的关系，添加反向链接
4. 建议归档位置（concepts/ 或 summaries/）
5. 提出 2-3 个深入问题供进一步探索

原始数据：
{{input}}

现有概念索引：
{{index}}
```

**步骤 3：建立索引系统**

- 主索引（index.md）：所有概念的目录
- 标签系统：统一的标签体系
- MOC（Map of Content）：主题地图，导航入口

**步骤 4：健康检查**

每月运行一次：

- 检测孤立页面（无入链也没出链）
- 发现矛盾信息
- 识别热门主题（高连接度概念）
- 建议新文章主题

### 6.3 第三阶段：智能问答与输出（1-2 个月）

**目标**：实现与知识库的自然对话

**步骤 1：配置问答模式**

**长上下文方案**（适合 <100万 token 知识库）：

- 使用 Claude 4 Opus 或 Gemini 2.5 Pro
- 加载整个 wiki 目录
- 直接提问，让 LLM 自主检索

**RAG 方案**（适合 >100万 token 知识库）：

- 部署向量数据库（Pinecone, Weaviate, Milvus）
- 建立 embedding 管道
- 配置检索参数（top-k, threshold）

**步骤 2：多模态输出**

配置输出模板：

- **报告模板**：Markdown，包含执行摘要、详细分析、参考资料
- **幻灯片模板**：Marp 格式，支持 CSS 主题
- **可视化模板**：Python matplotlib / Observable Plot

**步骤 3：建立反馈循环**

每次查询后：

- 评估答案质量
- 将高质量输出归档回 wiki
- 更新索引和概念链接
- 记录常见查询模式，优化索引

### 6.4 第四阶段：进阶优化（3-6 个月）

**目标**：构建领域专属 AI 助手

**步骤 1：合成数据生成**

- 从知识库提取问答对
- 生成推理链（chain-of-thought）数据
- 创建多轮对话数据集

**步骤 2：模型微调**

- 选择基础模型（Qwen3, Kimi2.5, DeepSeek）
- 使用 LoRA 进行领域微调
- 评估微调效果（对比基线）

**步骤 3：部署专属模型**

- 本地部署：Ollama, llama.cpp, vLLM
- 云端部署：Hugging Face Inference, Replicate
- API 封装：FastAPI, LangServe

**步骤 4：持续迭代**

- 监控模型性能
- 定期用新数据重新微调
- 收集用户反馈，改进数据生成策略

## 七、常见问题与避坑指南

### 7.1 数据质量陷阱

**问题**：知识库内容不清晰或缺乏标注，导致 AI 理解错误。

**解决方案**：

- 摄取前进行数据清洗
- 统一元数据格式（YAML frontmatter）
- 建立标签规范，强制一致性

### 7.2 过度依赖预训练模型

**问题**：直接使用通用 LLM，未结合领域知识，效果不佳。

**解决方案**：

- 始终提供上下文（即使使用长上下文模型）
- 定期用领域数据微调或至少进行 few-shot 提示
- 建立领域特定的提示词模板

### 7.3 忽视安全合规

**问题**：本地部署虽提升可控性，但仍需遵守数据安全法规。

**解决方案**：

- 了解《数据安全法》《个人信息保护法》要求
- 敏感数据脱敏后再入知识库
- 建立访问日志和审计机制

### 7.4 架构过度设计

**问题**：一开始就追求完美的系统，导致无法启动。

**解决方案**：

- 从简单开始，先跑通 MVP
- 使用"足够好"的工具，而非"最好"的工具
- 迭代优化，而非一次性完美

### 7.5 信息孤岛

**问题**：知识库与实际工作流脱节，成为"僵尸库"。

**解决方案**：

- 将知识库集成到日常工作工具（Slack, 飞书, Notion）
- 建立"知识库优先"文化：有问题先查库
- 定期回顾和更新，保持活跃

## 八、未来展望：AI 知识管理的演进方向

### 8.1 上下文窗口的持续扩大

**预测**：2026-2027 年，200万-500万 token 的上下文窗口将成为标配。

**影响**：

- 个人知识库可以完全加载到上下文中
- RAG 的必要性进一步降低
- 真正的"终身知识库"成为可能

### 8.2 多模态知识库

**趋势**：知识库不再限于文本，而是融合：

- **视频**（自动转录 + 关键帧提取）
- **音频**（播客、会议录音）
- **图像**（截图、照片、图表）
- **交互数据**（操作日志、传感器数据）

**技术**：Gemini 3.0 Pro 已经支持视频输入，未来 LLM 将能直接"观看"和"聆听"知识。

### 8.3 主动式知识管理

**演进**：从"被动查询"到"主动推送"

- AI 主动发现知识缺口，建议学习路径
- 基于当前任务，自动推荐相关知识
- 预测未来需求，提前准备信息

### 8.4 协作式知识网络

**愿景**：个人知识库 → 团队知识库 → 组织知识网络

- 知识在团队间流动和融合
- AI 协助发现跨领域连接
- 形成集体智慧，超越个体之和

### 8.5 神经符号融合

**前沿方向**：结合神经网络（模糊推理）和符号系统（精确逻辑）

- 用 LLM 进行知识提取和关联
- 用知识图谱进行结构化存储和推理
- 实现可解释、可验证的 AI 知识管理

## 九、结语：知识工作者的 AI 时代

Andrej Karpathy 的这篇推文，看似只是分享一个个人工作流，实则揭示了一个深刻趋势：AI 正在重新定义知识工作的本质。

从"写代码"到"管理知识"，Karpathy 的转变代表着 AI 应用范式的成熟——当工具足够强大时，人类的角色从"执行者"转变为"策展者"和"提问者"。

**核心启示：**

- **AI 是伙伴，不是替代品**：LLM 负责整理和维护，人类负责提问和判断
- **简单优于复杂**：长上下文窗口让 RAG 变得可选，简单架构往往更有效
- **持续积累胜过一次性完美**：知识库是活系统，需要持续喂养和迭代
- **输出即输入**：将洞察回流到知识库，形成自我增强的飞轮

**行动建议：**

不要等待"完美的工具"或"足够的数据"。今天就开始：

1. 选择一个工具（Obsidian 或飞书）
2. 摄取第一篇文档
3. 让 AI 生成第一份摘要
4. 提出第一个问题

正如 Karpathy 所说：

> "我认为这里有机会出现一个令人难以置信的新产品，而不是一堆脚本的拼凑。"

在那一刻到来之前，我们可以用现有的工具，开始构建自己的第二大脑——一个由 AI 增强的、不断进化的、真正属于你的知识库。
